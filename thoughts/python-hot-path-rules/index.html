My golden rules when writing Python

#python  #performance  #data-structures  #algorithms 

Contents

  • No Python loops on the critical path
  • Batch everything — repeated calls compound
  • The data structure is the algorithm
  • Don’t allocate on the hot path
  • Replace if/elif chains with dispatch tables
  • Parse at the boundary — carry clean types inward
  • Keep I/O out of the hot path
  • Conclusion

E very Python performance problem I’ve debugged traces back to the same mistake: Python doing work it should have handed off.

The interpreter overhead is predictable and manageable — the mistake is letting it accumulate inside the section of code that runs a million times. These are some of the learnings or rather rules of thumb I use.

No Python loops on the critical path#

Python’s for loop dispatches bytecode on every iteration. On millions of items in a request handler or data pipeline, it becomes the bottleneck — not because the work is expensive, but because the interpreter is doing it one item at a time.

Python’s built-ins — sum(), any(), all(), map() with a C-level callable — execute their inner loops in C. Not syntactic sugar over a Python loop; a different execution path.

total = 0
for x in values:    # interpreter overhead per element
    total += x

total = sum(values) # one Python call, C handles the rest

When the built-ins don’t cover the operation, that’s the signal to push the work into a compiled layer. The goal is to express what you want computed, not how to iterate.

Batch everything — repeated calls compound#

Every external call carries overhead beyond the work itself: a DB query has a round-trip, a cache GET has socket overhead, even a C extension call has setup cost. The mistake is paying that overhead N times to do N units of work.

The unit of work should be a batch, not an item.

The classical N+1 query problem:

# one round-trip per order — latency scales with row count
for order in orders:
    customer = db.query(
        "SELECT * FROM customers WHERE id = %s", order.customer_id
    )
    enrich(order, customer)

# one round-trip, join in the db where it belongs
customers = {
    c.id: c for c in db.query(
        "SELECT * FROM customers WHERE id = ANY(%s)",
        [o.customer_id for o in orders],
    )
}
for order in orders:
    enrich(order, customers[order.customer_id])

Same principle elsewhere: Redis pipeline over individual GETs, executemany over looped execute, bulk API calls over individual POSTs. Accumulate, then dispatch once.

The data structure is the algorithm#

Slow Python code almost always has the wrong data structure rather than the wrong algorithm. The structure determines the cost of every operation on it — and that cost multiplies across every call in the hot path.

A list used for membership checks is O(n) per lookup. A set is O(1). That difference is irrelevant at ten items and catastrophic at ten thousand.

blocked_ids = get_blocked_ids()       # returns a list

if user_id in blocked_ids: ...        # O(n) scan

blocked_ids = set(get_blocked_ids())  # one-time conversion

if user_id in blocked_ids: ...        # O(1) hash lookup

The rest of the standard library follows the same logic: deque for queues, Counter instead of manual dict accumulation, bisect for sorted search, defaultdict to eliminate the if key not in d guard. Each one exists because the naïve alternative has a worse complexity profile.

If a list is being used for lookups and it’s not trivially small, it’s wrong. Convert it to a set at construction time, not at query time — the bug is in where it’s built, not where it’s checked.

Don’t allocate on the hot path#

Every object constructed inside a tight loop — a list, a dict, a formatted string — is a heap allocation. Python’s GC reclaims them, but reclaims happen at unpredictable times and pause execution. The correct response is to stop generating garbage in the first place.

Generators are the lightweight version: constant memory, values produced on demand.

def active_users(records):
    return [r for r in records if r.active]   # full list in memory

def active_users(records):
    return (r for r in records if r.active)   # one item at a time

String concatenation in a loop allocates a new object on every +=. "".join(parts) allocates once at the end. On any non-trivial input the difference is an order of magnitude.

Replace if/elif chains with dispatch tables#

A dispatch table is a dict that maps keys to functions. Instead of branching logic deciding what to call, you look up the callable directly and invoke it. The name comes from how OS kernels route system calls.

A long if/elif chain is sequential lookup that gets worse as cases are added. It also couples extension to editing — a new case means touching the chain.

def handle(event_type, payload):
    if event_type == "order.created":
        on_order_created(payload)
    elif event_type == "order.updated":
        on_order_updated(payload)
    elif event_type == "payment.failed":
        on_payment_failed(payload)
    # every new handler extends this chain

HANDLERS: dict[str, Callable] = {
    "order.created":  on_order_created,
    "order.updated":  on_order_updated,
    "payment.failed": on_payment_failed,
}

def handle(event_type, payload):
    if handler := HANDLERS.get(event_type):
        handler(payload)

O(1) lookup, handlers registered independently of dispatch logic, and the walrus operator keeps the check and assignment on one line without calling .get() twice.

Parse at the boundary — carry clean types inward#

Parsing is cheap in isolation. It becomes expensive when it happens inside processing logic on every call rather than once at ingestion. Every layer that re-parses the same field is doing redundant work in a place that’s hard to profile.

Parse raw input once, at the point it enters the system. Every function downstream receives typed values.

# parsing runs on every call to process()
def process(raw: dict):
    user_id = int(raw["user_id"])
    ts = datetime.fromisoformat(raw["timestamp"])
    ...

# parse at the edge, internal functions see typed models
@dataclass
class Event:
    user_id: int
    timestamp: datetime
    payload: dict

    @classmethod
    def from_raw(cls, raw: dict) -> "Event":
        return cls(
            user_id=int(raw["user_id"]),
            timestamp=datetime.fromisoformat(raw["timestamp"]),
            payload=raw["payload"],
        )

def process(event: Event): ...
This is precisely what Pydantic does well — parse and validate at the boundary, clean model objects everywhere internal. The performance cost of Pydantic is at ingestion, not in your hot path.

Keep I/O out of the hot path#

print(), logger.info(), file writes, HTTP calls — all block the current thread. Inside a loop, the aggregate latency from logging alone can exceed the actual processing time. This is the performance bug that surprises people most because it looks harmless in isolation.

for record in records:
    result = process(record)
    logger.info("processed", id=record.id, status=result.status)  # blocks every iteration

results = [process(r) for r in records]
logger.info("batch complete", count=len(results), failures=sum(1 for r in results if r.failed))

When per-record observability is genuinely required, buffer log entries and flush after the loop, or push them to an async handler that writes off the main thread. The hot path should never wait on I/O.

Conclusion#

Most of these are a one-line change. A list becomes a set, a query moves outside a loop, a log call batches.You begin to see immediate impact.

The prerequisite is knowing where your hot path actually is. cProfile for function-level attribution, py-spy for profiling in production without code changes, line_profiler for line-level precision. Measure first — these rules tell you what to fix once you know where to look.

Fast Python isn't about avoiding Python — it's about knowing exactly where to stop using it carelessly.
