My golden rules when writing Python

#python  #performance  #data-structures  #algorithms

P ython is slow. That sentence alone ends most performance conversations before they start — but it’s the wrong frame. The real question is: which parts of Python are slow, and how do you keep them off the critical path?

Most performance-sensitive Python — numerical computing, trading systems, ML inference — is fast not because Python got faster, but because the hot path barely runs Python at all. The rules below are what that looks like in practice.

No Python Loops on the Critical Path#

For loops, while loops, list comprehensions, map() — they all carry the Python interpreter overhead on every iteration. At scale, that overhead compounds.

The alternative is to express iteration as array operations. NumPy, pandas, and PyTorch all execute their inner loops in C or Fortran. When you write a + b on two arrays, no Python loop runs.

# slow: Python iterates
result = [x * 2 for x in data]

# fast: C iterates
result = np.array(data) * 2
This rule only matters on the hot path — the code that runs millions of times. Looping over config at startup costs nothing.

Use Typed Arrays, Not Python Objects#

Python lists, dicts, and tuples carry per-element overhead: reference counting, type tags, pointer indirection. A list of a million floats is a million separate Python objects in memory.

Typed arrays eliminate that. A np.float64 array of a million elements is a contiguous block of 8MB. Iteration is cache-friendly, SIMD-vectorisable, and doesn’t touch the Python heap.

# a million Python objects, scattered in memory
prices = [101.5, 102.0, 101.8, ...]

# a contiguous block of float64 — one allocation
prices = np.array([101.5, 102.0, 101.8, ...], dtype=np.float64)
dtype=np.float32 halves memory and often improves cache hit rate, at the cost of precision. Worth benchmarking in ML workloads.

The same principle applies across the stack: prefer array.array over list, pd.Categorical over object columns, typed dataclasses over plain dicts.

Batch Calls — Every Python Call Has a Cover Charge#

Every call from Python into a C extension (NumPy, a database driver, a network client) has overhead: argument validation, GIL coordination, stack setup. One call on a million elements is fast. A million calls on one element each is slow.

Prefer batching, operation fusion, or doing everything in one shot.

This shows up most in database access:

# one round-trip per row — never do this
for row_id in ids:
    cursor.execute("SELECT * FROM orders WHERE id = %s", (row_id,))

# one round-trip for all rows
cursor.execute("SELECT * FROM orders WHERE id = ANY(%s)", (ids,))

The same pattern applies to API calls, file writes, and vectorised operations. When in doubt, accumulate and flush — don’t call in a loop.

No Allocations on the Hot Path#

Every list(), dict(), np.zeros(), or string format in a tight loop is a memory allocation. Allocations take time, and eventually the garbage collector reclaims them — also taking time, and often at the worst moment.

Preallocate buffers outside the loop and reuse them:

# allocates a new array every call
def compute(data):
    result = np.empty(len(data))
    result[:] = data * 2
    return result

# preallocate once, reuse across calls
buffer = np.empty(MAX_SIZE)

def compute(data, out=buffer):
    np.multiply(data, 2, out=out)
    return out[:len(data)]
NumPy’s out= parameter on most ufuncs writes results into an existing array instead of allocating a new one. Use it on any path that runs more than a few times per second.

Replace Branches with Masks and Lookup Tables#

if/else inside a loop forces sequential execution — the CPU can’t pipeline across an unpredictable branch. More importantly, it means Python is deciding something per element, which defeats the point of array operations.

Replace conditional logic with boolean masks or lookup tables:

# branch per element
result = []
for x in data:
    result.append(x * 1.1 if x > 100 else x * 0.9)

# vectorised: no Python branch
mask = data > 100
result = np.where(mask, data * 1.1, data * 0.9)

Lookup tables work for discrete mappings. Instead of a chain of if statements over a category code, build a dict or array indexed by the code and hit it once per element.

Conversions Are Poison#

list to np.array, .astype(np.float64), .tolist(), .values on a DataFrame — every one of these copies data. A conversion on the hot path means you’re allocating and filling a new buffer for every batch.

The fix is to get the data into the right shape once, at the boundary where it enters your system, and keep it in that shape throughout.

# wrong: converting on every inference call
def predict(raw_input: list):
    arr = np.array(raw_input, dtype=np.float32)  # allocation + copy
    return model(arr)

# right: convert at the edge, pass arrays internally
def ingest(raw_input: list) -> np.ndarray:
    return np.array(raw_input, dtype=np.float32)

def predict(arr: np.ndarray):
    return model(arr)
dtype mismatches are the silent killer here. Passing float64 to a model that expects float32 triggers an implicit .astype() copy. Always check dtypes at the boundary.

No I/O or Logging Inside the Hot Path#

print(), logging.debug(), file writes, and network calls have latency measured in microseconds to milliseconds. Inside a loop running at MHz, that’s a full stop.

The pattern is to collect, not emit. Accumulate metrics or log entries in a buffer; flush them asynchronously or at the end of a batch. Structured logging frameworks like structlog make this straightforward.

# blocks every iteration
for event in stream:
    process(event)
    logger.info("processed", id=event.id)  # I/O on hot path

# collect first, flush after
results = []
for event in stream:
    results.append(process(event))

logger.info("batch complete", count=len(results))  # once

Conclusion#

None of these rules require a different language. They require knowing where Python ends and the fast path begins — and keeping the interpreter out of the inner loop.

Profile first. cProfile, py-spy, and line_profiler will show you exactly which lines are costing you. The rules above tell you what to do once you’ve found them.

Fast Python isn't about avoiding Python — it's about knowing where to stop using it.
